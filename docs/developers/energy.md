# Daily energy

Daily energy records one private per-user row per local calendar date with the
user's active and resting energy expenditure in kcal. It intentionally mirrors
the backend shape of [`body_measurements`](database.md#auxiliary-domains--body-measurements-journal-and-daily-energy):
manual entry can provide either metric independently, and later iOS HealthKit
import will upsert only missing dates.

## Table contract

`public.daily_energy` stores:

| Column | Contract |
| --- | --- |
| `id` | UUID primary key, generated by Postgres. |
| `user_id` | Required owner FK to `auth.users(id)`, `ON UPDATE CASCADE ON DELETE CASCADE`. |
| `energy_on` | Required local calendar date for the expenditure totals. Unique with `user_id`. |
| `active_kcal` | Nullable `numeric(7,2)` active energy. If present, `0 <= active_kcal < 30000`. |
| `resting_kcal` | Nullable `numeric(7,2)` resting energy. If present, `0 <= resting_kcal < 30000`. |
| `notes` | Optional free-text note. |
| `created_at`, `updated_at` | Server timestamps; `updated_at` is maintained by the shared trigger function. |

Constraints are part of the public contract:

- `daily_energy_user_date_key` allows only one row per
  `(user_id, energy_on)`.
- `daily_energy_at_least_one_value_check` requires at least one of
  `active_kcal` or `resting_kcal`.
- `daily_energy_active_kcal_range_check` and
  `daily_energy_resting_kcal_range_check` reject negative values and values
  `>= 30000`.
- `daily_energy_user_date_idx` supports user/date listing in newest-first
  order.

## Hasura contract

The user-role GraphQL surface uses countable entry names instead of treating
"energy" as a plural noun:

- Queries: `dailyEnergyEntries`, `dailyEnergyEntry`,
  `dailyEnergyEntriesAggregate`.
- Mutations: `insertDailyEnergyEntry` / `insertDailyEnergyEntries`,
  `updateDailyEnergyEntry` / `updateDailyEnergyEntries`, and
  `deleteDailyEnergyEntry` / `deleteDailyEnergyEntries`.
- Columns are camel-cased as `userId`, `energyOn`, `activeKcal`,
  `restingKcal`, `createdAt`, and `updatedAt`.

Permissions follow pattern A in [`permissions.md`](permissions.md): rows are
private per user, `user_id` is set from `X-Hasura-User-Id` on insert, and
`user_id` is not in the user-role insert/update column allowlist. Clients may
insert or update only `energy_on`, `active_kcal`, `resting_kcal`, and `notes`;
they may select owner/timestamp columns for display and cache ownership checks.

Hasura `numeric` values must be normalized at client boundaries before math or
display. Use the same string-or-number numeric handling already used for body
measurements and nutrition nutrient columns when parsing `activeKcal` and
`restingKcal`.
